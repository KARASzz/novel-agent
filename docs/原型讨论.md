### 一、 核心架构：流水线作业

我们将整个过程拆解为三个步骤，前两步由大模型（GLM-5/MiniMax）负责，最后一步由 Python 脚本负责。

1. ​**数据层**​：大模型输出 ​**JSON 数据**（纯数据，无格式）。
2. ​**逻辑层**：Python 脚本读取 JSON，进行逻辑校验、字数统计、格式清洗。
3. ​**表现层**：Python 脚本按照模板生成最终文件（Markdown/Word/PDF/Fountain格式）。

### 二、 具体落地步骤

#### 步骤 1：定义标准数据结构

不要让模型直接写剧本文本，而是让它填“表”。你需要定义一个 JSON Schema。

```hljs
JSON 格式定义：
{
  "script_meta": {
    "title": "剧名",
    "episode": "集数",
```

**为什么这样做？**   
JSON 是程序员发明的数据格式，极其严格。GLM-5 和 MiniMax 的代码能力都很强，输出 JSON 的准确率远高于输出带格式的文本。

#### 步骤 2：编写格式化渲染脚本

这是你的核心竞争力。你用 Python 写一个脚本，定义“工业化标准”。

**Python 脚本示例 (**​**​`script_formatter.py`​**​ **)：**

python

![](assets/image-20260329204859-0cdu2vq.svg)复制

```hljs
import json

def format_script(json_data):
    output_lines = []
    
    # 1. 处理头部信息
    meta = json_data['script_meta']
    output_lines.append(f"剧本标题：{meta['title']}")
    output_lines.append(f"第 {meta['episode']} 集")
    output_lines.append("=" * 30 + "\n")

    # 2. 遍历场景
    for scene in json_data['scenes']:
        # 场景标题格式：△ S001 [豪宅-客厅] (日)
        scene_header = f"△ {scene['scene_id']} [{scene['location']}] ({scene['time']})"
        output_lines.append(scene_header)
        output_lines.append("-" * 30)

        # 3. 遍历剧情点
        for plot in scene['plot_points']:
            if plot['type'] == 'action':
                # 动作描写格式：【动作】(3s) 他猛地拍桌子，眼神冰冷。
                line = f"【动作】({plot['duration_sec']}s) {plot['content']}"
            else:
                # 台词格式：人物(情绪)：台词内容
                char_info = f"{plot['character']}({plot['emotion_tag']})"
                line = f"{char_info}：{plot['content']}"
            
            # 4. 工业化校验逻辑（代码的强项）
            # 例如：如果台词超过30字，强制添加 [需拆分] 标记
            if plot['type'] == 'dialogue' and len(plot['content']) > 30:
                line += " [警告：台词过长，建议拆分]"

            output_lines.append(line)
        
        output_lines.append("\n") # 场景间空行

    return "\n".join(output_lines)

# 使用示例
if __name__ == "__main__":
    # 假设这是模型输出的 JSON 数据
    model_output_json = """
    {
      "script_meta": {"title": "逆袭风云", "episode": "1"},
      "scenes": [
        {
          "scene_id": "S001", "location": "宴会厅", "time": "夜",
          "plot_points": [
            {"type": "action", "content": "主角林萧推门而入，全场寂静。", "duration_sec": 5, "emotion_tag": null, "character": null},
            {"type": "dialogue", "content": "今天，我要拿回属于我的一切！", "duration_sec": 3, "emotion_tag": "霸气", "character": "林萧"}
          ]
        }
      ]
    }
    """
    
    data = json.loads(model_output_json)
    formatted_text = format_script(data)
    print(formatted_text)
```

### 四、 方案优势总结

1. ​**绝对的格式统一**：空格、标点、换行完全由代码控制，模型偶尔的“手抖”完全不会影响最终排版。
2. ​**可扩展性强**​：如果你想改剧本格式（比如把场景号从 `S001`​ 改成 `Scene-1`），只需改一行 Python 代码，不用重新训练或调试模型。
3. ​**可嵌入业务逻辑**​：比如“自动计算一场戏的总时长”、“自动检测是否符合平台规范（如每集必须有一个反转）”，这些用 Prompt 很难写，用 Python 的 `if/else` 非常简单。

### 总结

**不要试图用 Prompt 告诉模型“怎么排版”，而是用 Prompt 让模型“输出数据”，用 Python 脚本“排版”。**

这是目前实现**短剧剧本工业化**最稳健、最高效的技术路径。这样，你用 GLM-5 或 MiniMax 都能跑出完全一样格式的剧本，彻底消除了模型间的差异。

### 步骤二：Python 脚本做“工业级校验”

拿到上面的 JSON 后，Python 脚本可以做很多大模型做不好、或者是容易做错的事情：

```hljs
import json

def render_and_validate(json_data):
    total_duration = 0
    output_text = ""
    
    for scene in json_data['scenes']:
        # --- 工业排版 ---
        output_text += f"\n【场景 {scene['scene_id']}】 {scene['location']} - {scene['time']}\n"
        output_text += "-" * 40 + "\n"
        
        for plot in scene['plots']:
            # 累计时长
            total_duration += plot['duration_sec']
            
            # 格式化输出
            if plot['type'] == 'action':
                output_text += f"（{plot['content']}）\n"
            else:
                # 强制格式：人物(情绪)：台词
                output_text += f"▷ {plot['character']}（{plot['emotion']}）：{plot['content']}\n"
    
    # --- 商业逻辑校验 ---
    output_text += "\n" + "=" * 40 + "\n"
    output_text += f"[系统统计] 本段总时长：{total_duration}秒\n"
    
    if total_duration > 120:
        output_text += "[系统警告] ⚠️ 时长超标，建议削减非主线台词！\n"
    
    return output_text

# 模拟读取大模型生成的 JSON

# 执行渲染
```

### 步骤三：利用“正则”预处理极端混乱的草稿

如果你的草稿格式极其混乱（比如有的用括号，有的用冒号，有的只有对话没有名字），建议在送给大模型前，先用 Python 做一次​ **“粗清洗”** 。

这能降低大模型的理解难度，提升准确率。

**Python 预处理脚本：**

python

![](assets/image-20260329204859-7nacdng.svg)复制

```hljs
import re

def pre_clean_draft(text):
    # 1. 统一全角/半角标点
    text = text.replace(":", "：").replace("(", "（").replace(")", "）")
    
    # 2. 识别缺失人名的对话（假设短对话独占一行）
    # 这是一个简化的逻辑：如果行首没有“【”或“（”，且有冒号，可能是对话
    lines = text.split('\n')
    cleaned_lines = []
    for line in lines:
        if '：' in line and not line.startswith(('（', '【', '△')):
            # 尝试提取人名
            parts = line.split('：', 1)
            if len(parts[0]) <= 4: # 假设人名不超过4个字
                # 这是一个有效对话格式，保留
                cleaned_lines.append(f"▷ {line}")
            else:
                # 可能是长难句，加个标记给大模型处理
                cleaned_lines.append(f"[待确认] {line}")
        else:
            cleaned_lines.append(line)
    
    return '\n'.join(cleaned_lines)
```

## 二、配套转化脚本（完整可运行）

python

![](assets/image-20260329204859-277ihgi.svg)复制

```hljs
import re
from datetime import datetime

class ScriptFormatter:
    """短剧剧本格式化工具"""
    
    def __init__(self):
        self.speed_chars_per_min = 200  # 语速：每分钟200字
        self.current_scene_id = 0
        self.total_duration = 0
        
    def parse_draft(self, draft_text):
        """解析草稿文本"""
        lines = draft_text.strip().split('\n')
        
        result = {
            "meta": {},
            "scenes": []
        }
        
        current_scene = None
        
        for line in lines:
            line = line.strip()
            if not line:
                continue
            
            # 1. 解析剧本信息（=== 开头）
            if line.startswith('===') or line.startswith('剧名') or line.startswith('集数'):
                self._parse_meta(line, result["meta"])
                continue
            
            # 2. 解析场景标记（### 开头）
            if line.startswith('###'):
                # 保存上一个场景
                if current_scene:
                    result["scenes"].append(current_scene)
                
                # 开始新场景
                self.current_scene_id += 1
                scene_info = self._parse_scene_header(line)
                current_scene = {
                    "scene_id": f"S{self.current_scene_id:03d}",
                    "location": scene_info["location"],
                    "time": scene_info["time"],
                    "plots": []
                }
                continue
            
            # 3. 解析剧情内容
            if current_scene:
                plot = self._parse_plot_line(line)
                if plot:
                    current_scene["plots"].append(plot)
        
        # 保存最后一个场景
        if current_scene:
            result["scenes"].append(current_scene)
        
        return result
    
    def _parse_meta(self, line, meta_dict):
        """解析元信息"""
        if '剧名' in line:
            meta_dict['title'] = line.split('：')[-1].strip()
        elif '集数' in line:
            meta_dict['episode'] = line.split('：')[-1].strip()
    
    def _parse_scene_header(self, line):
        """解析场景头"""
        # 格式：### 豪宅-客厅 日
        content = line.replace('###', '').strip()
        parts = content.split()
        
        location = parts[0] if len(parts) > 0 else "未指定"
        time = parts[1] if len(parts) > 1 else "未指定"
        
        return {"location": location, "time": time}
    
    def _parse_plot_line(self, line):
        """解析单行剧情"""
        # 情况1：【动作描写】
        if line.startswith('【') and line.endswith('】'):
            content = line[1:-1]
            duration = self._calc_duration(content, is_dialogue=False)
            return {
                "type": "动作",
                "character": None,
                "content": content,
                "duration_sec": duration
            }
        
        # 情况2：人名：台词（包括独白）
        if '：' in line:
            parts = line.split('：', 1)
            character = parts[0].strip()
            content = parts[1].strip()
            
            # 检测是否是独白
            is_monologue = '独白' in character or '内心' in character
            duration = self._calc_duration(content, is_dialogue=True)
            
            return {
                "type": "独白" if is_monologue else "台词",
                "character": character.replace('（独白）', '').replace('（内心）', ''),
                "content": content,
                "duration_sec": duration
            }
        
        # 情况3：普通动作描写（没有括号的）
        # 假设不含冒号的行都是动作
        duration = self._calc_duration(line, is_dialogue=False)
        return {
            "type": "动作",
            "character": None,
            "content": line,
            "duration_sec": duration
        }
    
    def _calc_duration(self, text, is_dialogue):
        """计算时长（秒）"""
        char_count = len(text.replace('，', '').replace('。', '').replace('！', '').replace('？', ''))
        
        if is_dialogue:
            # 台词：按语速计算
            duration = (char_count / self.speed_chars_per_min) * 60
        else:
            # 动作：按描述长度估算（稍慢）
            duration = (char_count / 100) * 60
        
        return round(max(duration, 1))  # 最少1秒
    
    def format_script(self, data):
        """格式化输出"""
        output = []
        
        # 头部信息
        meta = data.get("meta", {})
        output.append("=" * 50)
        output.append(f"剧名：{meta.get('title', '未命名')}")
        output.append(f"集数：{meta.get('episode', '未指定')}")
        output.append(f"生成时间：{datetime.now().strftime('%Y-%m-%d %H:%M')}")
        output.append("=" * 50)
        output.append("")
        
        # 遍历场景
        for scene in data["scenes"]:
            # 场景标题
            output.append(f"△ {scene['scene_id']} [{scene['location']}] ({scene['time']})")
            output.append("-" * 50)
            
            scene_duration = 0
            
            # 遍历剧情
            for plot in scene["plots"]:
                scene_duration += plot["duration_sec"]
                
                if plot["type"] == "动作":
                    output.append(f"  （{plot['content']}）[{plot['duration_sec']}s]")
                elif plot["type"] == "独白":
                    output.append(f"  ▷ {plot['character']}（独白）：{plot['content']} [{plot['duration_sec']}s]")
                else:
                    output.append(f"  {plot['character']}：{plot['content']} [{plot['duration_sec']}s]")
            
            output.append("")
            output.append(f"  [场景时长：{scene_duration}秒]")
            output.append("")
        
        # 统计信息
        output.append("=" * 50)
        output.append(f"[总时长：{self.total_duration}秒 / {self.total_duration/60:.1f}分钟]")
        output.append(f"[场景数：{len(data['scenes'])}]")
        output.append("=" * 50)
        
        return '\n'.join(output)
    
    def process(self, draft_text):
        """主流程：草稿 -> 成品"""
        # 计算总时长
        data = self.parse_draft(draft_text)
        for scene in data["scenes"]:
            for plot in scene["plots"]:
                self.total_duration += plot["duration_sec"]
        
        return self.format_script(data)
```

### 1.3 具体写作方法（逐条落地的写法）

#### 1）先定“核心误会 + 核心期待”

竖屏短剧文章里提得很明确：

- 男频常见：​**身份误会**——所有人都以为男主是废物，其实他是隐世神豪/战神html5.qq.com
- 女频常见：​**情感误会**——白月光、替身、误会导致的虐恋html5.qq.com

做法：

- 一上来先写一句话：

  - “这部剧的核心误会是：所有人以为女主是拜金女，其实她是隐藏的首富之女。”
- 再写一句话：

  - “观众的期待是：看她何时身份揭晓，打脸那些欺负她的人。”

#### 2）再选一个爆款框架做“骨架”

AI 竖屏短剧实践里提到，他们会对“闪婚、离婚后、虐恋、战神”等框架做拆解，形成可复用模板html5.qq.com。

举例：

- “闪婚框架”公式：

  - 女主因故和看似普通的男主闪婚 → 婚后男主消失 → 女主在职场重逢高冷霸总 → 其实两人是夫妻 → 身份揭晓、打脸众人html5.qq.com

你可以：

- 选 1–2 个对标爆款（红果上同类型的 Top）
- 把它的框架拆成：开篇误会、前 10 集期待、第 10 集卡点、后 20 集反转
- 然后在自己的故事里，用“新人物 + 新场景”做微创新toutiao.com。

#### 3）前三集：黄金 6 分钟必须完成三件事

根据“三集定律”html5.qq.com+1：

- 第一集：

  - 开场 3–5 秒要有画面冲击（吵架、车祸、强吻、退婚等）
  - 交代主角身份和起点（弱、惨、被误解）
  - 引出主线事件（复仇、离婚、闪婚、穿越等）
- 第二集：

  - 让误会加深、冲突升级
  - 第一次小反转/打脸
- 第三集：

  - 把观众情绪拉到某个“临界点”，为第 4–10 集的卡点做准备

#### 4）每集的“卡点”和“钩子”

- 爆款短剧写作强调：每一集都要有“卡点意识”，结尾要有悬念或紧张感toutiao.com。
- AI 竖屏短剧实践总结：

  - 卡点本质是​**情绪的临界点**——把愤怒、期待、委屈压到极限，再突然反转或悬住html5.qq.com。
  - 两种黄金卡点模式：

    1. 主线/人物关系反转处：比如“虐到极致，即将反转的那一刻”html5.qq.com
    2. 期待解开处：开篇抛出悬念，卡在“即将揭晓身份的前一秒”html5.qq.com

做法：

- 每集结尾，都问自己三个问题：

  1. 观众现在最期待什么？
  2. 这集把期待拉得有多高？
  3. 我卡在“期待即将被满足的前一秒”了吗？

#### 5）台词与节奏：短、爆、情绪前置

爆款短剧写作方法里提到：

- 一集字数 450–550，不能大段堆台词toutiao.com
- 台词要短，在符合人设的同时要有爆发力toutiao.com

AI 竖屏短剧实践里总结：

- “所有情绪表达在台词里”，台词要直白、有力、情绪外化html5.qq.com
- 情绪前置：先给情绪，再说事情；用标签化语言强化人设html5.qq.com

你可以：

- 每句台词都检查：

  - 是否超过 2 行？超过就删。
  - 是否只是“说明情况”，没有情绪？改成“情绪 + 行动”。
  - 是否有口头禅、说话方式让人记得住？

## 二、提示词设计的核心理念

结合 AI 写短剧的实战文章csdn.net+1和爆款短剧写作方法toutiao.com，可以归纳出​**四个设计理念**：

### 理念 1：先给“结构约束”，再给“内容自由”

- 不要只说“写一个都市爽剧”，而是：

  - 先规定题材、风格、误会类型、卡点位置
  - 再让大模型在框架内“填空”

AI 写短剧的实战文章也强调：Prompt 要包含四个核心要素csdn.net+1：

1. 题材与风格
2. 人物设定
3. 剧情大纲
4. 格式要求

这四个，其实就是“结构约束”。

### 理念 2：把“写作方法”翻译成“Prompt 规则”

比如前面提到的写作方法，可以直接变成 Prompt 条件：

- “每集必须有开篇冲突 + 中段反转 + 结尾钩子”
- “台词不要超过 2 行，要有情绪爆发力”
- “前三集必须完成：背景、人物关系、主线剧情”
- “第 10 集要设计一个付费卡点，卡在情绪临界点”

这些规则直接写进 Prompt，比一句“帮我写个爆款短剧”靠谱得多。

### 理念 3：把“输出”结构化为 JSON，方便后端脚本处理

你已经有了“草稿模板 → JSON → Python 渲染”的前端流程，那么提示词最好：

- 让大模型输出​**结构化 JSON**，而不是直接输出成片剧本
- JSON 里包含：场景号、地点、时间、人物、情绪标签、时长估算、是否为卡点等字段

这样就能复用你现在的 Python 流水线。

### 理念 4：分阶段生成，而不是一次性“写全剧”

AI 竖屏短剧实践里是分阶段做的html5.qq.com+1：

- 先生成创意（核心误会 + 多个冲突方案）
- 再搭“集纲”（1–20 集骨架）
- 再填充每集具体场次和台词

你也应该：

- Step 1：Prompt 生成“创意 + 人物小传 + 集纲”
- Step 2：按集生成场次/台词
- Step 3：用 Python 做格式化、统计时长、卡点标注

```hljs
class ScriptFormatter:
    def __init__(self):
        self.speed_chars_per_min = 200
        self.current_scene_id = 0
        self.total_duration = 0
        
    def validate_format(self, script_text):
        """校验剧本是否符合工业格式"""
        errors = []
        warnings = []
        lines = script_text.strip().split('\n')
        current_scene = None
        
        for i, line in enumerate(lines):
            line = line.strip()
            if not line:
                continue
                
            # 检查索引格式
            if line.startswith('△') and ']' in line:
                current_scene = line
                # 检查索引格式：△ S001 [内/外] [日/夜/晨/昏]
                if not re.match(r'^△\s+S\d{3}\s+\[内/外\]\s+\[日/夜/晨/昏\]$', line):
                    errors.append({
                        "line": i+1,
                        "error_type": "索引格式错误",
                        "description": f"索引格式不符合标准：{line}"
                    })
            
            # 检查环境描述
            elif line.startswith('【') and line.endswith('】'):
                # 环境描述应该包含地点和人物信息
                if '在哪里' not in line and '都有谁' not in line:
                    warnings.append({
                        "line": i+1,
                        "warning_type": "环境描述不完整",
                        "description": "环境描述缺少地点或人物信息"
                    })
            
            # 检查动作格式
            elif line.startswith('（') and line.endswith('）'):
                # 动作描述应该有内容
                if len(line) < 10:  # 简单判断，实际可以更复杂
                    warnings.append({
                        "line": i+1,
                        "warning_type": "动作描述过短",
                        "description": "动作描述内容不足"
                    })
            
            # 检查对白格式
            elif '：' in line:
                parts = line.split('：', 1)
                if len(parts) == 2:
                    character = parts[0].strip()
                    dialogue = parts[1].strip()
                    if len(dialogue) < 5:  # 简单判断，实际可以更复杂
                        warnings.append({
                            "line": i+1,
                            "warning_type": "对白过短",
                            "description": f"对白内容不足：{dialogue}"
                        })
        
        # 检查钩子
        if script_text.count('钩子') < 1:
            warnings.append({
                "line": "全局",
                "warning_type": "缺少钩子",
                "description": "剧本缺少结尾钩子"
            })
        
        return {
            "is_valid": len(errors) == 0,
            "errors": errors,
            "warnings": warnings,
            "format_score": max(0, 100 - len(errors) * 10 - len(warnings) * 5)
        }
```

## 为什么选择JSON进行校验？

### 1. **机器可读性**

- JSON是结构化数据格式，程序可以直接解析
- Markdown是自然语言，需要复杂的解析逻辑才能提取信息
- 在自动化流程中，JSON能被Python等语言轻松处理

### 2. **标准化和一致性**

- JSON有明确的语法规则，不会出现歧义
- Markdown格式可能因人而异，难以标准化
- JSON能确保每次校验的输出格式一致

### 3. **丰富的元数据支持**

- JSON可以包含详细的错误信息：错误类型、行号、描述、建议等
- Markdown难以组织如此结构化的信息
- 例如：`{"error_type": "缺少▲标记", "line": 15, "description": "动作/环境缺少▲标记"}`

### 4. **易于集成和自动化**

- 可以轻松集成到现有的Python流水线中
- 便于后续的自动化修复和报告生成
- 支持批量处理和存储校验结果

### 5. **可扩展性**

- JSON可以轻松添加新的校验规则和字段
- 便于版本控制和更新
- Markdown格式难以扩展和修改

```hljs
        document += f"链接: {content['url']}\n"
        document += f"发布时间: {content['published_date']}\n\n"
        document += f"### 关键内容\n{content['cleaned_content']}\n\n"
        document += "---\n\n"
    
    return document
```
