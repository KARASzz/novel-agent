# 1-《前置中台 v1.0 模块蓝图》

行，这次我给你的是**能落系统的施工图**，不是空中飘 prompt。

先砍你一句狠的：  
你这个 LTM 草案方向是对的，但有个**接口认知硬伤**——**​`UpdateProfileSchema`​**​ **不是日常经验回写桶，它是“画像模板字段结构”的升级接口**。  
日常写入主用：

- ​**​`AddMemory`​**：加记忆，支持从对话提炼，也支持直接写入记忆内容
- ​**​`SearchMemory`​**：召回相关长期记忆
- ​**​`UpdateMemory`​**：修正已有记忆
- ​**​`UpdateProfileSchema`​**：只在你要新增/修改“作者画像字段”时动它

按阿里云官方文档，记忆库这条线是成立的：记忆库支持 `AddMemory / SearchMemory / ListMemory / UpdateMemory / DeleteMemory`​，画像模板单独维护，`memory_library_id`​ 不传可走默认库，`userId`​ 用于记忆隔离。你这个“单人专属跨剧本经验库”的定位，**非常适合接进去**。  
参考：阿里云官方文档 [记忆库](https://help.aliyun.com/zh/model-studio/memory-library)、[长期记忆 API 2.0](https://help.aliyun.com/zh/model-studio/long-term-memory-2-0)、[OpenClaw 记忆插件配置说明](https://help.aliyun.com/zh/model-studio/modelstudio-memory-for-openclaw)

# 《前置中台 v1.0 模块蓝图》

## 一、先定总原则：不是“三库”，你现在要升级成“三库一脑一影子”

### 1. 三库

#### A. 市场时效库

存：

- 近 30/60/90 天平台动态
- 扶持计划
- 题材信号
- 合规与侵权案例
- 形态分流信号（真人 / AI / 混合）

不存：

- 作者个人经验
- 当前项目最终决策

#### B. 行业常识库

存：

- 长期有效的叙事规律
- 结构方法论
- 红果通用过稿标准
- 题材常模

不存：

- 时效性很强的热点
- 个人偏好

#### C. 项目知识库

存：

- 当前项目的赛道判断
- 概念分支
- GoT 图谱
- 风险包
- 准入决定
- 版本演化记录

不存：

- 跨剧复用的作者长期经验

---

### 2. 一脑：作者专属 LTM

这是你的**个人写作脑**。  
专门记：

- 你历史上真正有效的钩子打法
- 你经常踩的结构坑
- 你容易高估的新鲜度类型
- 你擅长的反转风格
- 你在不同赛道上的命中率差异

**不记当前项目的人设细节和剧情事实。**   
那是项目库的事，不是 LTM 的事。  
你要是把“第 7 部剧男主姓什么”也塞进 LTM，这脑子三个月就臭了。

---

### 3. 一影子：本地 LTM 审计镜像

这个非常重要，别省。

作用：

- 做写入前去重
- 做写入后审计
- 做云端不可用时兜底
- 记录“哪条经验什么时候写进去、由什么证据支持”

建议只做**结构化影子快照**，不做全量替代。

# 二、LTM 在前置中台里的正确位置

LTM 不是末端外挂，应该是**前置中台的旁路决策脑**。

## 读路径：前置读取

在这些节点读：

1. **立项初筛前**

   - 读你的强项/弱项
2. **赛道分流前**

   - 读你在哪种赛道容易误判
3. **概念竞技前**

   - 读你历史高 ROI 钩子与失败模式
4. **对抗验证前**

   - 读你最常见的伪创新、自嗨、过度复杂化问题
5. **准入评审前**

   - 读你历史上“看起来很新其实转化差”的误判案例

## 写路径：延迟写回

在这些节点写：

1. **对抗验证完成后**

   - 只写“已验证的可复用教训”
2. **样稿完成并通过人工验收后**

   - 写“结构经验”和“风格经验”
3. **上线后有真实反馈时**

   - 写“市场校正经验”
4. **项目复盘结束后**

   - 写“最终稳定经验”

### 不要在什么阶段写

- 不要把临时脑洞直接写进 LTM
- 不要把尚未证实的“我感觉这次会火”写进去
- 不要把当前项目的专属设定当成长久规律写进去

# 五、LTM 的推荐画像模板设计

这个你一定要收住，**别一上来建 40 个字段**。  
v1.0 先建 10 个字段，够用了。

## 推荐 `profile_schema` 字段

1. ​`best_hook_families`

   - 作者高命中开篇钩子类型
2. ​`strongest_lanes`

   - 作者最擅长赛道
3. ​`weakest_lanes`

   - 作者最易误判赛道
4. ​`preferred_conflict_drivers`

   - 作者最稳的冲突发动机
5. ​`common_failure_modes`

   - 高频失败模式
6. ​`overused_cliches`

   - 容易重复使用的陈词滥调
7. ​`novelty_bias`

   - 作者对“新鲜度”的误判倾向
8. ​`rewrite_preference`

   - 作者偏好的改稿方式
9. ​`format_strength`

   - 真人 / AI / 混合哪种更适合你
10. ​`risk_sensitivity`

- 对合规/版权/授权风险的敏感度

### 什么时候用 `UpdateProfileSchema`

只有两种情况：

- 你发现画像字段不够表达你的创作特征
- 连续多个项目证明，你需要新增一个长期维度

比如你以后要单独跟踪：

- ​`ai_rights_sensitivity`
- ​`series_ip_bias`

这时候再改 schema。  
**不是每次复盘都去动 schema。**

# 九、最后一句结论

你这个 LTM 模块，**不该塞在最后当装饰件**。  
它应该被定义为：

> **前置中台的“作者专属决策脑”，与 RAG 的“客观知识脑”并列存在。**

一句最清楚的话送你：

- **RAG 负责告诉系统：红果现在客观上在发生什么**
- **LTM 负责告诉系统：你这个作者，主观上最会赢什么、最容易死什么**

这两脑不分开，你的系统永远只是“会写”，不会“会判断”。

如果你要，我下一步可以继续直接给你出：

1.  **《前置中台 v1.0 数据结构清单》**   
   每个输出包的字段名、类型、必填项
2.  **《LTM 记忆写回规则 v1.0》**   
   什么该写、什么不该写、怎么去重、怎么防污染

你这一步已经不是 prompt 工程了，宝贝。  
这是在搭你自己的**剧本操作系统内核**。
